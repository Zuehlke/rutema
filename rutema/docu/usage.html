Rutema Usage

Even the best intentions...

...will not save you if your client has to do all the work.

Rutema is flexible, almost too flexible. You can pick and choose which parts to use and in practical day-to-day usage this has been done. One example is a tool that generates Excel documents presenting requirements coverage using the base parser.

The problem with all this flexibility is that you probably don't know what to do with it to begin with.
So Rutema comes with a few bonuses to get you started.

Installation is very easy:

gem install rutema
Make sure you install all required gems.

Out of the box

Installing the gem adds three rutema related commands to the system:
  • rutemax is the command that executes tests based on a configuration
  • rutemah is a tool to create simple reports froma database of rutemax test results
  • rutemaweb is a web frontend for any database of rutemax test results

Simple usage

From version 0.4.2 rutema comes with a sample called distro_test that illustrates the minimal usage rutemax can be put to. Theres a whole chapter about it.

Advanced usage

Rutema's base parser does not produce runnable scenarios.
The MinimalXML parser is intended as an example and it serves as a simple tool that will let the developers create the first tests, a critical mass that will lead to the creation of a test DSL
It is really not practical to maintain many test specifications where the paths and parameters are hardcoded in command elements.
The best way, actually the only way, to take full advantage of Rutema's capabilities is to write your own parser, one that will speak your project's DSL and make life easier for testers and developers alike.
Rutema tries to make it easy, but there is no going around it.

So after the installation comes the planning stage, where one defines the DSL and implements the parser.
In planning the parser, you inadvertedly influence the configuration of you project. Rutema's configuration allows you to pass values such as tools and paths, plus other contextual data (i.e. version information) to the various Rutema components.

The same can be said for what comes at the tail end of rutema's work: reporting. Here, the basic report produced by rutema is OK but still limited to the last run with no history.
Just like with the parser, you can write your own reporter, but in this case we can provide you with more meaningful reporter implementations.
A reporter that saves test results in a database using Active Record is available, as is an email reporter.

Let's test something

Finally, after the parser is implemented and the configuration is done, you can use Rutema's test execution tool. 

Usage: rutemax [options] command|filename
Options:
    -d, --debug                      Turns on debug messages
    -c, --config FILE                Loads the configuration from FILE
    -l, --log FILE                   Redirects the log output to FILE
        --check                      Runs just the check test
    -V, --version                    Displays the version
    -h, -?, --help                   This text
The commands are:
    all                            Runs all tests
    attended                       Runs all attended tests
    unattended                     Runs all unattended tests
You can also provide a specification filename in order to run a single test

Shooting ourselves in the foot!

In all honesty, experience shows that each project has different requirements on what a tool like Rutema should do.
And the basic interface never seems enough, once you get used to doing things automagically.
The hottest issue is executing groups of tests according to one criterion or another, usually based on an attribute of the specification or the scenario.

Experience shows that classification attributes within the XML are better avoided, for maintainability if not for any other reason.
There is also the question of scale for a file based system like the one shown here.
In order to execute queries over the sum of specifications, they all need to be parsed. Real-world data shows it will not scale well past the 1000 specifications mark using the REXML based parser. Using a C based XML parser should significantly increase speeds though.

The use of a database for storing specifications has at this point been consciously discarded as the primary goal was to make creation of the specifications easy and not create another GUI tool with it's own editor.
A text format is also a lot easier to extend/change without breaking a lot of things while versioning can be easily delegated to a version control system making Rutema's implementation that much simpler.

Ofcourse, nobody keeps you from replacing the parser with a DB access layer, replacing Rutema::Coordinator to provide a richer interface and adding an editor that updates the database for good measure. We expect you to donate the code to the project when everything is up, running and well tested :).
